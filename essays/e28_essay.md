One of the best characteristics of programming is its accessibility, where anyone can take any approach vector coding. Everyone is different in their ability and what they are comfortable with, and each could develop a process of writing code that is the mos efficient for them. One of my example of this is from the IntelliJ editor. An external terminal emulator of a Linux machine could be used to install add-ons like npm, but the IDE also offers its own terminal for anyone who prefers Windows. It’s very difficult to not find several alternatives to your workflow. However, this characteristic is an two-edged sword, since people brought with them different ways of writing code. To correct and make code easier to read, the dreaded coding standards would have to dampen this benefit of programming.
	Perhaps its because of the nature of learning software engineering, where you can use any approach you like (online resources, books, CC’s, and universities), that people dislike the limitation of programming in one way. Before my exposure to coding standards, programming requires less restrictions and formatting rules than other forms of technical writing. Research papers, with their templates, page formatting, and citations, is very different from the almost-free-flow statements and brackets placements. One could break an expression into multiple lines or reduce an if statement into one lines, and the compiler would still process the code. This possibility of writing code in any desired would be limited by coding standards.  As a student, I could see the dissimilarity between the free-form learning and the strict programming rules being disorientating for some.
	When I got past the learning curve, however, the limiting coding standard began to show its possibility. My code became more readable, and the structure friendlier towards searching and debugging. Applying my observations to the bigger picture, coding standards allows for a unified formatting for coders. Everyone now have the same coding style. This is a huge time saver for team-based coding, taking the time for trying to understand the code and push it towards actually programming. The standard is limiting for individuals, but it allows for easier cooperation and sharing of code. A coding standard became a necessary “evil” to maintain the best characteristic of programming. Anyone could approach programming in whichever way that is suitable to them, but that wouldn’t be a true accessibility if each person’s code can’t be understood by everyone.
